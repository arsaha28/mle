Overview

Generative AI with LangChain

A hands-on guide — run each file, read what prints, move to the next.

What is an LLM?

A Large Language Model is a neural network trained on vast text. It predicts the most likely next token — but at scale this produces reasoning, summarisation, code, and more.

The full pipeline we'll build
Prompt → LLM → Parser → Tools → Memory → RAG → Agent
Key tools in this tutorial
  • LangChain — wires together prompts, models, parsers, and retrievers
  • LCEL — LangChain Expression Language; compose with the | pipe operator
  • LangGraph — builds stateful, looping, multi-agent workflows as a graph
  • OpenAI — provides the underlying LLM (gpt-4o-mini) and embeddings
  • FAISS / Chroma — vector databases for semantic search
Before you start
  • Copy .env.example → .env and add your OPENAI_API_KEY
  • Run pip install -r requirements.txt
  • Run files in order — each builds on the last
Foundations

Basic LLM Calls

Your first conversation with a model.

01_basic_llm.py
Core idea

ChatOpenAI is the model wrapper. You send it text, it returns an AIMessage. Three modes: invoke (one shot), stream (token by token), batch (many prompts at once).

Key points
  • temperature=0 → deterministic. temperature=1+ → creative
  • invoke() blocks until the full response arrives
  • stream() yields chunks — use for live UI updates
  • batch() sends multiple prompts in parallel — faster than a loop
  • Response is an AIMessage; read text via .content
Watch for in the code
  • How stream() uses end="" to print without newlines between chunks
  • The difference in output style between temperature 0 and 1.2
Foundations

Message Types

Chat models speak in roles, not raw strings.

02_messages.py
Core idea

Every chat model call is a list of messages, each with a role. The model reads the full list and decides what to say next. Getting roles right is the foundation of everything else.

The four roles
  • SystemMessage — sets persona/rules. The model follows this silently
  • HumanMessage — the user's turn
  • AIMessage — a past model response; inject to give it "memory"
  • ToolMessage — the result of a tool call, fed back to the model
Watch for in the code
  • How injecting an AIMessage makes the model "remember" Alice's name
  • MessagesPlaceholder — a slot in a template filled with a list at runtime
  • The ToolMessage pattern: model asks → tool runs → result injected → model responds

Prompt Templates

Reusable prompts with variables.

03_prompt_templates.py
Core idea

Hard-coded prompts are brittle. Templates define the structure once with {variables} and fill them at runtime — making prompts composable and testable.

Template types
  • PromptTemplate — plain string with {vars}
  • ChatPromptTemplate — list of role-message pairs with {vars}
  • SystemMessagePromptTemplate — typed system message with variables
  • HumanMessagePromptTemplate — typed human message with variables
  • Partial templates — pre-fill some vars now, leave others for later
Watch for in the code
  • base_template.partial(role="doctor") locks one variable, leaves others open
  • from_template() shorthand vs explicit constructor

Prompt Strategies

How you ask matters as much as what you ask.

04_prompt_strategies.py
Core idea

The same model gives dramatically different answers depending on how the prompt is framed. These strategies are your toolkit for improving output quality.

Six strategies — side by side
  • Zero-shot — just a task description, no examples. Fast, less precise
  • One-shot — one example teaches the output format
  • Few-shot — multiple examples; the model generalises the pattern
  • Few-shot chat — same idea but using Human/AI message pairs
  • Chain-of-Thought — "Let's think step by step" forces reasoning before the answer
  • Self-Consistency — run CoT 3× at high temperature; majority vote wins
Watch for in the code
  • The bat-and-ball problem — classic CoT demo. Notice how reasoning improves accuracy
  • FewShotPromptTemplate vs FewShotChatMessagePromptTemplate — same idea, different format

LCEL Basics

Compose components with the pipe operator.

05_lcel_basics.py
Core idea

LCEL chains are built by piping Runnables together with |. Each component's output becomes the next one's input. The chain itself is also a Runnable — so it has invoke, stream, and batch.

Key points
  • Standard chain: prompt | llm | parser
  • Data flows left → right automatically
  • StrOutputParser converts AIMessage → plain str
  • Multi-step: reshape output with a lambda between steps
  • chain.get_graph().print_ascii() visualises the pipeline
Watch for in the code
  • The two-step chain — explain in English, then translate to French
  • The ASCII graph output — shows every node and edge

Runnable Primitives

Advanced building blocks for complex data flows.

06_lcel_runnables.py
Core idea

Beyond the basic pipe chain, LCEL offers primitives for splitting, merging, routing, and transforming data — all composable with |.

Five primitives
  • RunnablePassthrough — pass input unchanged; useful to keep original data alongside LLM output
  • RunnableParallel — run multiple chains on the same input simultaneously
  • RunnableLambda — wrap any Python function as a chainable Runnable
  • RunnableBranch — route to different chains based on a condition
  • itemgetter — extract one key from a dict cleanly inside a chain
Watch for in the code
  • RunnableParallel pros/cons run at the same time — note both keys in the result dict
  • RunnableBranch — the fallback (last arg) fires when no condition matches

Output Parsers

Turn LLM text into typed Python objects.

07_output_parsers.py
Core idea

Raw LLM output is always a string. Parsers convert it into the type your application needs — and some parsers inject format instructions into the prompt automatically.

Six parsers
  • StrOutputParser — AIMessage → str. Use this 90% of the time
  • CommaSeparatedListOutputParser — str → list[str]
  • JsonOutputParser — str → dict. Model must output valid JSON
  • PydanticOutputParser — str → validated Pydantic model. Best for structured data
  • DatetimeOutputParser — str → datetime
  • XMLOutputParser — str → parsed XML dict
Watch for in the code
  • get_format_instructions() — auto-injects the expected format into the prompt
  • Pydantic parser validates types — year is always an int, rating always a float

Document Loaders

Get your data into LangChain.

08_document_loaders.py
Core idea

Every loader produces a list of Document objects. Each document has page_content (the text) and metadata (source, page number, etc.).

Five loaders
  • TextLoader — entire file as one Document
  • PyPDFLoader — one Document per page; metadata includes page number
  • CSVLoader — one Document per row; column values in content
  • DirectoryLoader — all files matching a glob pattern in a folder
  • WebBaseLoader — scrapes and parses a URL into one Document

  • PDF gives one doc per page; CSV gives one doc per row — shape differs by loader

Text Splitters

Break large documents into embeddable chunks.

09_text_splitters.py
Core idea

LLMs have a context limit. Splitters cut documents into overlapping chunks so each fits within the limit. chunk_overlap ensures context isn't lost at boundaries.

Five splitters
  • CharacterTextSplitter — splits on one separator. Simple but rigid
  • RecursiveCharacterTextSplitter — tries paragraph → line → word → char. Use this by default
  • TokenTextSplitter — counts tokens not chars; matches LLM limits exactly
  • MarkdownHeaderTextSplitter — splits on # headings; keeps hierarchy in metadata
  • HTMLHeaderTextSplitter — splits on <h1>/<h2> tags
Watch for in the code
  • Markdown/HTML splitters put heading names in metadata — great for citations
  • Compare chunk counts between splitters on the same document

Embeddings & Vector Stores

Convert text to vectors; find similar text instantly.

10_embeddings_vectorstores.py
Core idea

An embedding is a list of numbers that encodes meaning. Texts about similar topics have vectors that are close together. A vector store indexes these vectors and retrieves the nearest ones to a query.

Key concepts
  • OpenAIEmbeddings — calls the OpenAI API to embed text (text-embedding-3-small)
  • FAISS — Facebook's fast in-memory index. No server needed. Save/load to disk
  • Chroma — persistent local vector store with a simple API
  • similarity_search(query, k=3) — returns the k nearest documents
  • Cosine similarity — the maths behind "how similar are two vectors"
Watch for in the code
  • The manual cosine similarity section — see how scores differ for relevant vs irrelevant text
  • allow_dangerous_deserialization=True is required to reload a FAISS index from disk

Retrievers

Smarter ways to fetch relevant documents.

11_retrievers.py
Core idea

A retriever wraps a vector store with a clean .invoke(query) interface. Different strategies trade off between speed, diversity, and relevance.

Five retrievers
  • Basic — top-k by similarity. Fast, straightforward
  • MMR — Maximum Marginal Relevance. Penalises redundant results → more diverse
  • MultiQueryRetriever — LLM generates 3 query variants; results merged. Better recall
  • ContextualCompression — LLM strips irrelevant sentences from each chunk before returning
  • SelfQueryRetriever — LLM translates natural language into metadata filters automatically
Watch for in the code
  • MMR's lambda_mult — 0 = max diversity, 1 = max relevance
  • SelfQuery turns "page 2" in English into an actual metadata filter

Basic RAG

End-to-end retrieval-augmented generation.

12_rag_basic.py
Core idea

Instead of relying on training data, the model reads real documents at query time. This reduces hallucination and keeps answers current. All the pieces from 08–11 combine here.

The pipeline — step by step
  • Load — PyPDFLoader reads laptop_policy.pdf
  • Split — RecursiveCharacterTextSplitter chunks it
  • Embed + Store — FAISS indexes all chunks
  • Retrieve — top-4 chunks fetched per question
  • Prompt — chunks injected as context; LLM answers from context only
Watch for in the code
  • RunnableParallel fetches context and passes the question through simultaneously
  • The system prompt says "ONLY the context" — this prevents hallucination
  • Source chunk display at the end — shows exactly which pages were used

Advanced RAG

Conversational, multi-query, and reranked retrieval.

13_rag_advanced.py
Core idea

Basic RAG fails on follow-up questions and misses relevant chunks with poor phrasing. These three patterns fix those problems.

Three patterns
  • Conversational RAG — condense "What about that?" → standalone question using history, then retrieve
  • Multi-query RAG — generate 3 phrasings of the question, merge all results, deduplicate
  • Reranking — retrieve 8 chunks, score each with an LLM, keep the top 3
Watch for in the code
  • The condensation step — see how "Do they need a VPN?" becomes a full standalone question
  • Reranking adds LLM calls per chunk — slower but more precise
Memory & Tools

Conversation Memory

Give the model a sense of history.

14_memory.py
Core idea

LLMs are stateless — every call starts fresh. Memory is just storing past messages and injecting them into the next prompt. The challenge is controlling how much history to keep.

Three patterns
  • Full history — keep every message. Simple, but grows without limit
  • Sliding window — keep only the last N messages. Caps token usage
  • Summarisation — compress old messages into a running summary. Best of both
Watch for in the code
  • RunnableWithMessageHistory — auto-loads and saves history per session_id
  • trim_messages(strategy="last") — keeps the most recent N, always preserving SystemMessage
  • Summarisation class: when messages exceed window, old ones collapse into one summary message
Memory & Tools

Tools

Let the model take actions.

15_tools.py
Core idea

Tools extend what an LLM can do. The model reads tool descriptions and decides whether — and how — to call them. You define the tools; the model decides when to use them.

Four ways to define a tool
  • @tool decorator — simplest. Docstring becomes the description the model reads
  • StructuredTool — explicit name/description/function wiring
  • Pydantic args schema — type-safe validated inputs with field descriptions
  • BaseTool subclass — full control; good for complex or async tools
Watch for in the code
  • llm.bind_tools(tools) — tells the model what tools exist
  • The manual execution loop — model returns tool_calls; you run them; feed results back as ToolMessage
LangGraph

Graph Basics

Stateful workflows as nodes and edges.

16_langgraph_basics.py
Core idea

LangGraph models workflows as a directed graph. Each node is a Python function that reads from shared state and returns updates. Edges define which node runs next.

Core concepts
  • StateGraph — the builder. Takes a TypedDict as the state schema
  • Node — a function (state) → dict. Returns only the keys it changes
  • Edge — add_edge(A, B) means: after A, always go to B
  • START / END — built-in entry and exit points
  • compile() — validates the graph and returns a runnable app
Watch for in the code
  • Nodes return partial dicts — only update the keys they touch
  • print_ascii() — visualise the graph structure in the terminal
LangGraph

Conditional Routing

Branches, parallel fans, and retry loops.

17_langgraph_conditional.py
Core idea

Real workflows branch. A router function inspects state and returns the name of the next node. This unlocks if/else logic, parallel execution, and loops inside a graph.

Three patterns
  • Conditional edge — router returns a node name based on state. Technical vs general answers
  • Parallel branches — add two edges from START to run nodes simultaneously; merge at a join node
  • Retry loop — conditional edge returns back to a previous node until score threshold is met
Watch for in the code
  • add_conditional_edges(node, router_fn) — the router returns a node name, not a boolean
  • Two add_edge(START, X) calls — LangGraph runs them in parallel automatically
  • Loop: should_continue returns either "generate" or END

ReAct Agent

The foundational agentic loop.

18_langgraph_agent.py
Core idea

ReAct = Reason + Act. The agent loops: the LLM thinks → calls a tool → reads the result → thinks again → until it has a final answer. LangGraph makes this loop explicit and inspectable.

The loop
  • agent node — LLM with tools bound; returns either tool_calls or a final answer
  • tool_node — executes every tool call in the last message; appends ToolMessages
  • router — if last message has tool_calls → tool_node; else → END
  • add_messages reducer — appends new messages instead of replacing the list
Watch for in the code
  • The Annotated[list, add_messages] state field — this is what enables the message loop
  • The multi-step question — the agent calls list_length, then feeds that result into calculate

Advanced Patterns

Human-in-the-loop, supervisor agents, error recovery.

19_langgraph_advanced.py
Core idea

Production agents need to pause for humans, delegate to specialists, and recover from failures gracefully. These three patterns cover the most common real-world requirements.

Three patterns
  • Human-in-the-loop — interrupt() suspends the graph; resume after update_state() with the human's decision
  • Supervisor — a coordinator LLM routes each task to the best specialist (writer / analyst / coder)
  • Error recovery — wrap risky nodes in try/except; store error in state; route to a fallback handler
Watch for in the code
  • MemorySaver is required for interrupt() — it checkpoints state between invocations
  • Supervisor pattern: add_conditional_edges uses lambda s: s["next_agent"] as the router
  • Error recovery: conditional edge checks state.get("error") — empty string is falsy
